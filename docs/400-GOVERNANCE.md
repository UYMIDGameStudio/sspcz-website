# RFC-400: Engineering Governance & Success Criteria

## 1. Pull Request / Commit Red Lines
Any code changes—whether authored by humans or generated by AI Agents—MUST adhere to the following strict constraints to be merged:

1. **Zero Inline Styles**: `style="..."` is strictly forbidden in HTML markup.
2. **No Utility-Class Pollution**: Prefer semantic, component-level class names (e.g., `.editorial-quote`). Tailwind-style utility soup (e.g., `flex mt-4 text-blue`) is prohibited.
3. **DRY i18n**: Absolute zero duplicated HTML markup across language versions. Structural components must handle bilingual text injection dynamically.
4. **Zero-JS Baseline**: The core reading experience, navigation, and content consumption must function flawlessly with JavaScript disabled in the browser.

## 2. Component Boundaries (Single Responsibility)
- **Patterns (`src/patterns/`)**: Handle cross-page visual motifs (e.g., `<Marginalia>`, `<RunningHeader>`).
- **Layouts (`src/layouts/`)**: Define the document flow and grid (e.g., `<EditorialLayout>`).
- **Components (`src/components/`)**: Strictly stateless, scoped elements. A `<SectionMarker>` must only handle marking a section, not typography or external margins.

## 3. Quality Assurance
- **Lighthouse Score**: Performance, Accessibility, Best Practices, and SEO must simultaneously exceed 95.
- **Traceability**: Every visual decision (spacing, color tone, font size) MUST be traceable to a central Design Token (CSS Variable). No hardcoded magic numbers.
